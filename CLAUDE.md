## Project Overview

GaggiMate is a smart upgrade system for Gaggia espresso machines. It consists of two ESP32 firmware targets (display and controller) that communicate over BLE, plus a Preact web UI served from the device's SPIFFS filesystem.

## Build Commands

### Firmware (PlatformIO)

```bash
# Build specific environment
platformio run -e display
platformio run -e controller
platformio run -e display-headless

# Upload to device
platformio run -e display -t upload
platformio run -e display -t uploadfs   # Upload web assets to SPIFFS
```

## Architecture

### Two-Board Design

- **Controller board** (ESP32-S3): Drives relays, reads thermocouple (MAX31855), pressure sensor, manages PID. Source: `src/controller/`.
- **Display board** (LilyGo-T-RGB or similar): Runs LVGL UI, hosts the web server, implements all high-level logic. Source: `src/display/`. Communicates with controller over BLE.

BLE communication is handled by `lib/NimBLEComm/` with custom UUIDs. The display is the BLE client; the controller is the BLE server.

### Display Firmware Architecture

The display firmware uses an **event-driven plugin system** centered in `src/display/core/`:

- **`Controller`** (`core/Controller.h/cpp`): Top-level orchestrator. Owns the `PluginManager`, active process, and settings. Coordinates mode transitions (standby → brew → steam, etc.).
- **`PluginManager`** (`core/PluginManager.h`): Singleton managing plugin lifecycle (`setup()` / `loop()`). Plugins register event listeners via `pluginManager->on(eventId, callback)`.
- **`EventManager`** (`core/Event.h`): Typed event bus. `EventDataEntry` can hold int/float/string. Events fired with `EventManager::trigger()`.
- **`Settings`** (`core/Settings.h/cpp`): Wraps ESP32 `Preferences` API. Use `settings.batchUpdate(callback)` for multiple changes.
- **`ProfileManager`** (`core/ProfileManager.h`): Loads/saves espresso profiles from SPIFFS as JSON.

**Plugins** (`src/display/plugins/`) each implement a `Plugin` interface. Notable plugins:
- `WebUIPlugin`: Async HTTP + WebSocket server. All web API traffic flows through here.
- `BLEScalePlugin`: Connects to Bluetooth scales for volumetric dosing.
- `MQTTPlugin`, `HomekitPlugin`, `ShotHistoryPlugin`, `AutoWakeupPlugin`, `SmartGrindPlugin`.

**Process classes** (`core/process/`): `BrewProcess`, `SteamProcess`, `PumpProcess`, `GrindProcess` inherit from `Process`. They interpret `Profile` data (phases, targets, transitions) and emit control commands.

**Modes** (defined in `core/constants.h`):
```cpp
MODE_STANDBY = 0, MODE_BREW = 1, MODE_STEAM = 2, MODE_WATER = 3, MODE_GRIND = 4
```

### Web API (WebSocket)

The device hosts a WebSocket at `/ws`. Message format: JSON with a `tp` field for type.
- `req:*` — client request; `res:*` — device reply with matching `rid`.
- `evt:*` — device push (status, OTA progress, autotune).

Full spec: `docs/websocket-api.yaml` (AsyncAPI format).

`web/src/services/ApiService.js` implements the client with auto-reconnect and 30 s response timeout.

### Web Interface

Preact SPA with file-based routing (`web/src/pages/`). Major pages: Home (live dashboard), ProfileList/ProfileEdit, ShotHistory/ShotAnalyzer, Settings, Autotune, OTA, Scales. Shared components in `web/src/components/`.

### LVGL UI

UI screens and components under `src/display/ui/default/lvgl/` are generated by **SquareLine Studio** (source project in `ui/`). Avoid hand-editing generated screen files; make changes in SquareLine Studio and re-export. Custom logic hooks go in `src/display/ui/utils/`.

### Hardware Abstraction

`src/display/drivers/` provides a `Driver` base class with implementations for LilyGo-T-RGB, AMOLED, and Waveshare displays. The active driver is selected at compile time via build flags in `platformio.ini`.

## Data Formats

Espresso profiles are JSON files stored in `data/p/`. Schema: `schema/profile.json`. A profile has `phases`, each with `name`, `duration`, `temperature`, `pump` settings, and optional `target`/`transition` for advanced pressure/flow control.

Shot history schema: `schema/shot_history.json`. Shot notes schema: `schema/shot_notes.json`.

## Code Style

- **C++17**, LLVM-based clang-format: 4-space indent, 130-char line limit (see `.clang-format`).
- PascalCase for classes, camelCase for variables/methods, `#pragma once` for header guards.
- Web: Prettier + ESLint. See `web/.eslintrc.*` and project `prettier` config.

## CI

- **`check.yml`**: Runs `platformio check` (cppcheck) on every push/PR.
- **`build.yml`**: Triggered on git tags — builds all firmware + web assets and creates a GitHub release.
- Version is auto-generated from `git describe` by `scripts/auto_firmware_version.py`.
